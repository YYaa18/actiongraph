ActionGraph

把企业业务能力组织成可规划、可审批、可恢复的 Agent Action 图。

ActionGraph 让模型只负责理解目标，让 Java 业务系统定义可执行能力、参数契约和治理边界。两注解即可进入金路径，运行时负责 schema 绑定、确定性规划、人审挂起、补偿恢复与审计证据。

把 Agent 能做什么，定义成可审查的工程契约。

ActionGraph 面向已有核心业务系统的团队：把订单、续约、理赔、转账、审批等能力封装成 Action 图，让 Agent 能办事，同时保留企业系统需要的边界、证据和控制权。

在企业系统里，让 LLM 直接调 API 或写 SQL 通常不可控、不可复现，也难以审计。只做问答机器人又无法把流程办完。每个业务系统各自接 AI，则会重复建设 prompt、审批、审计、回滚和运维标准。

ActionGraph 选择工程化路线：应用只暴露经过审查的 Action，Goal schema 定义参数契约，规划器只看 condition/effect，运行时每一步都经过权限、人审、runtime guard 和补偿边界。LLM 可以解释目标，但不能直接操作业务状态。

LLM 直连 API计划不可复现，越权和重复副作用难以治理。
只有问答机器人能解释制度，不能稳定完成审批、下单、生成草稿等流程。
系统各自接 AIprompt、审计、回滚、人审和运维标准分散，长期成本高。

从自然语言到业务副作用，运行链路保持可复现。

运行时把目标解释、schema 自动绑定、确定性规划、逐步执行、人审/事件挂起、恢复和补偿串成一个闭环。每一步执行后都会重新规划，因此运行时事实变化可以自动改道。

1

解释目标

GoalInterpreter 把用户输入映射为 GoalType 与参数。LLM 解释器可用，规则解释器可降级。

2

绑定 schema

Goal schema record 自动接收 customerId、orderId 等参数并写入类型化 Blackboard；复杂装配再使用显式 Seeder。

3

确定性规划

GoapPlanner 只根据 Action 的 precondition 与 effect 搜索可达路径，不让 LLM 生成计划。

4

受控执行

GoapExecutor 按步骤执行，检查权限、runtime guard、金额规则与人审策略。

5

恢复与审计

挂起快照、外部事件、补偿栈、Trace 哈希链与 Console read model 形成可追踪证据链。

接入路径：两注解定义能力，一个门面发起运行。

金路径从主 starter、Goal schema、Action 注解和根门面 ActionGraph 开始。先让 start(...) 跑通，再按 L1-L4 逐步加入人审、自然语言、持久化和跨服务事件。

1 引入运行时 starter

用 BOM 锁定版本，引入主 starter，默认自动扫描 Spring Bean 上的 Action 与 Goal 注解。

2 声明目标契约与能力

用 @ActionGraphGoal(schema=...) 声明参数契约，用 @ActionGraphAction 声明可执行业务能力。

3 从根门面启动

业务代码只注入 ActionGraph，调用 start、chat 或 resume；运营面 API 按需单独启用。

  implementation(platform("com.actiongraph:actiongraph-bom:0.2.0-SNAPSHOT"))

  implementation("com.actiongraph:actiongraph-spring-boot-starter")
  implementation("com.actiongraph:actiongraph-llm-deepseek")
}

ActionGraph 正门保持窄。

业务发起面只有 start、chat、resume。常规目标由 schema 自动绑定，显式 Seeder 只处理多对象写入、业务实体映射或自定义 seed conditions。审批决策、事件投递、恢复扫描和 Console 查询属于运营面。

运行时能力围绕一个原则：模型不拥有最终裁决权。

框架让 Agent 能执行，但不把业务判断交给模型。schema 契约、规划、运行时校验、审批、补偿、持久化、审计与控制面都围绕“可控执行”设计。

确定性 GOAP 规划

只基于 Action 前置条件与效果搜索路径，计划可复现、可测试、可审计。

Schema 自动绑定

@ActionGraphGoal(schema=...) 自动把请求参数绑定为 record 并写入 Blackboard，复杂场景再显式扩展。

Runtime Guard

依赖真实业务值的判断放到执行前校验，保持规划器纯净，不污染 symbolic planning。

多级人工审批

Repository-backed review 支持审批链、stage index、回调幂等与原子 suspend/resume。

Saga 补偿

失败或审批拒绝时按已执行 Action 逆序补偿，降低孤儿副作用残留。

Trace 哈希链

TraceEvent 按 runId 形成链式 SHA-256 哈希，Console 可显示审计链完整性。

敏感数据脱敏

Trace 与审批预览可脱敏身份证、银行卡、手机号、邮箱和指定字段。

额度与风险策略

业务方提供 AmountExtractor 后，可按 action / 币种设置硬拒绝，并在进入人审时追加金额升级审批阶段。

治理逻辑进入执行路径，而不是事后补丁。

权限、脱敏、额度、人审、补偿、哈希链和恢复快照全部围绕同一个 runId 与 Trace 序列展开，方便企业控制台和审计系统追踪。

权限与额度

PermissionPolicy 与 ExecutionPolicyGuard 在执行前判定动作是否允许；金额规则可硬拒绝，并为进入人审的动作追加金额升级属性。

  • 按 action id 与 currency 配置 hard-limit / review-limit
  • AmountExtractor 由业务侧提供，不让框架猜业务金额
  • RiskBasedChainResolver 根据 amountEscalated 追加 amount-authorization 阶段
  • 规则权限与租户校验可在非 Spring 服务手工装配

人审挂起与恢复

requiresHumanReview 的 Action 创建审批任务并保存挂起快照，审批后从同一个 runId 原子 claim 恢复，避免重复副作用。

  • PENDING 只挂起，不补偿
  • APPROVED 推进阶段，最终阶段通过后继续执行
  • DENIED 终止并触发已执行 Action 的逆序补偿

审计与脱敏

Trace 事件先经过 DataMaskingPolicy，再写入哈希链。挂起快照保持无损恢复，不做脱敏。

  • Trace detail/data 与审批预览支持字段与正则脱敏
  • TraceChainVerifier 可识别断链与篡改
  • Console read model 可分页筛选并导出归档证据

生产契约覆盖持久化、恢复、安全与兼容性。

JDBC 持久化、Java 8 老系统、控制层 token、GET 重试、Blackboard 类型白名单和 snapshot version 都是发布契约的一部分。

JDBC persistence 启用 actiongraph.persistence.jdbc.enabled=true 后，Trace、SuspendedRun、Trace read model、Memory 和 Human Review repository 可切到 JDBC。
Blackboard allowlist 生产恢复挂起快照时建议配置 allowed-packages，在反序列化前拒绝不允许的类型。
Java 8 control plane actiongraph-control-plane-api 编译为 Java 8，无 Spring、JDBC、LLM、Servlet 或 JSON 依赖。
Safe GET retries 可对组件目录、人审查询和 Console 只读查询开启 GET 重试；会产生副作用的 POST 不自动重试。
控制面 shared-secret 不是企业 IAM。

内置 token verifier 负责 Header 名校验、未配置 shared-secret 时跳过读取、常量时间比较 token。生产身份认证、RBAC、租户权限、网络边界和审计策略仍应由接入系统、API 网关或企业安全平台负责。

场景主线

用户提出“为客户 C001 预约 80 万元的大额转账”。运行时需要确认参数、绑定 schema、规划可达路径、处理余额接口重试、触发限额升级审批、挂起等待人审、恢复同一 run，并最终给出可验证的 Trace 证据。

01 · Intent

解释器只确认目标与参数

自然语言入口负责把请求映射到 GoalType、customerId、amount 等业务参数；缺参时返回澄清问题。计划仍由 GOAP planner 根据 Action 图搜索，LLM 不直接生成执行步骤。

  • 输入：客户、金额、业务意图。
  • 产物：Goal + schema record + seed conditions。
02 · Contract

启动前校验 Action 图

框架在应用启动或注册阶段检查 condition/effect 是否闭合。拼写错误、命名空间不匹配和悬空 Action 会尽早暴露，避免把不可达路径留到生产运行时才发现。

  • 检查：缺失条件、悬空动作、相近 effect 诊断。
  • 价值：把“AI 能走哪些路”变成可审查的工程契约。
03 · Execution

每一步执行后重新规划

余额实时接口失败时，Action 的重试策略先处理短暂异常；当 runtime guard 判断核心系统不可用时，规划器会根据当前事实改道到快照余额链路。业务编排不需要把每个 fallback 手写成流程分支。

  • 运行策略：maxAttempts、backoff、runtime guard。
  • 改道条件：Blackboard 事实变化后触发下一轮 planner 选择。
04 · Review

高风险动作挂起，而不是继续猜测

80 万元超过 review-limit 后，风险链路在原审批链上追加 amount-authorization。运行进入 SUSPENDED_PENDING_REVIEW，保存挂起快照与补偿栈；审批通过后以同一 runId 原子认领恢复。

  • 人审链：checker-review、authorization、amount-authorization。
  • 并发语义：重复 resume 只有一个请求能成功 claim。
05 · Evidence

审计证据随运行产生

Trace 事件先经过脱敏策略，再进入哈希链。Console read model 可以展示运行列表、Trace 时间线、审计链验证结果和导出文件；挂起快照保留恢复所需原值，不与展示层脱敏混淆。

  • 审计对象：Trace sequence、hash chain、review attributes。
  • 展示边界：审批预览和 Console 输出脱敏，恢复快照保持语义完整。
06 · Policy

硬限额在副作用发生前拦截

当金额超过 hard-limit，Policy 在执行草稿创建前拒绝动作，后续副作用不会发生。review-limit 只负责进入人审后的金额升级，不替代业务权限、租户校验或企业 IAM。

  • 拒绝点：Action 执行前。
  • 治理边界：ActionGraph 提供策略挂点，业务系统保留最终裁决。
07 · Durability

进程崩溃，副作用零孤儿

开启 durability 后，每个 Action 成功即写检查点，执行前登记执行意图（WAL）。故障注入场景模拟进程在高风险窗口中断——草稿已在远端创建、检查点未落。重启后的恢复器认领异常中断的运行记录：意图未清，该步按「结果未知」先补偿再重做，随后续跑进审批、照常挂起。真实输出：草稿创建成功×2、恢复性补偿×1 → 净草稿恰好 1 份；51 条 Trace 事件跨崩溃、恢复与审批连续编号，哈希链验证通过。

  • 恢复语义：in-flight 步与超时同一哲学——结果未知，补偿＋重做实现等效恰好一次。
  • 运维含义：编排服务可进行滚动发布；实测每步检查点开销 0.054 ms。
