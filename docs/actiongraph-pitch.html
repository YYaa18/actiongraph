为什么现在必须做

AI 进入业务系统，最大问题不是“能不能答”，而是敢不敢执行

01

LLM 直接操作系统

让模型自己拼 SQL、自己调 API，路径不可复现，幻觉不可控，审计说不清。生产环境不会接受这种风险。

02

只做问答助手

能解释制度、能查知识，但不能替员工把流程跑完。价值停在“咨询台”，进不了核心业务链路。

03

各系统各自接 AI

每个团队重复做 prompt、审批、审计、回滚。成本乘以系统数，风险也乘以系统数，标准无法沉淀。

我们需要的不是“更聪明的聊天框”，而是一个把 AI 执行力关进安全边界里的基础设施。
从已有 Java 方法变成可规划能力

01 REGISTER
Spring 扫描注解业务类不实现框架接口，starter 把 Bean 方法适配成 Action。
02 SEARCH
Planner 只看条件图中文 preconditions/effects 同样参与确定性规划。
03 EXECUTE
每步执行后重规划guard、审批、失败补偿都发生在运行期。

Action 白名单

只允许执行注册过的业务 Action。LLM 不能自由写 SQL、不能临时发明 API、不能跳过流程。

权限 / 租户策略

统一策略层在执行前判定 subject、tenant、riskLevel。拒绝即终止并触发补偿。

挂起不补偿 拒绝才补偿 补偿失败升级为 FAILED_COMPENSATION_INCOMPLETE runId / trace seq 跨恢复续接
为什么它值得做成平台

它不是替代工作流，也不是普通 Agent：它是业务 Action 图谱执行层

能力维度 直接接 LLM 工作流引擎 外部 Agent 平台 ActionGraph
执行路径可复现不可控可以取决于平台确定性搜索
新场景接入成本各自重复要画流程要适配平台注册 Action 即成图
异常改道能力靠模型猜需预画分支黑盒每步重规划
审批与补偿自研补偿自研不统一框架原生
企业数据边界容易外溢内部供应商绑定业务逻辑留在内部 Java 系统
审计可解释性弱流程级平台级目标 / 计划 / 每步 / 补偿全记录

boolean 策略允许创建报价(RenewalEligibility eligibility) {

void 作废报价草稿(QuoteDraft draft) { quoteService.voidDraft(draft.quoteId()); }

真实场景验证清单

进入业务系统前，先用一个真实场景验证效率、安全、可审计

场景准备

低风险高频链路

草稿生成、申请发起、内部审批类场景最适合作为首个验证对象，既能体现效率，也能控制业务风险。

工程联调

审批与持久化

跑通挂起、审批、恢复、拒绝补偿、失败补偿，安全团队检查审计链与权限策略。

灰度观测

小范围真实用户

采集操作耗时、接入人天、审批拦截率、审计完整率，为后续推广范围提供数据依据。

验证指标

单流程耗时下降 50% 以上；接入成本不超过 3 人天；审批拦截率、补偿成功率、审计完整率均达到 100%。

输出物

真实运行日志、Trace 审计样本、补偿演练记录、接入成本统计和风险问题清单。

战略价值

ActionGraph 的本质：把 AI 能力从“项目制”升级为公司级基础设施

每个系统注册自己的业务能力图谱，全公司共享一个安全、可审计、可恢复的 AI 执行底座。接入的系统越多，能力图谱越大，AI 能替员工完成的事就越多。

近期

首个真实场景落地验证，完成内部 Maven 发布、中文接入文档、审批系统对接。

中期

扩展到 2~3 个业务系统，沉淀公司统一 Action 设计规范和安全策略模板。

长期

形成内部能力图谱市场：业务能力可发现、可复用、可组合，新流程由图谱自动装配。

汇报结论：已具备进入真实场景验证的工程基础
