为什么现在必须做

AI 进入业务系统，最大问题不是“能不能答”，而是敢不敢执行

01

LLM 直接操作系统

让模型自己拼 SQL、自己调 API，路径不可复现，幻觉不可控，审计说不清。生产环境不会接受这种风险。

02

只做问答助手

能解释制度、能查知识，但不能替员工把流程跑完。价值停在“咨询台”，进不了核心业务链路。

03

各系统各自接 AI

每个团队重复做 prompt、审批、审计、回滚。成本乘以系统数，风险也乘以系统数，标准无法沉淀。

我们需要的不是“更聪明的聊天框”，而是一个把 AI 执行力关进安全边界里的基础设施。
ActionGraph 的核心分工

AI 理解目标，图搜索决定路径，业务服务执行真实动作

01 INPUT
自然语言目标“帮客户 C001 准备续约报价”
02 INTERPRET
LLM Goal Interpreter只抽取 goalType 和参数，不能生成计划
03 PLAN
确定性 GOAP Planner基于 precondition / effect 搜索 Action 图
04 GOVERN
策略与人工审批权限、租户、风险、人工 review 统一拦截
05 EXECUTE
业务系统执行调用已有 Java Service，失败按 Saga 补偿
每步执行后重新规划 高风险动作自动挂起 runId / trace seq 跨恢复续接 Blackboard 类型化数据

一句话演示口径

ActionGraph 把“AI 想做什么”和“系统允许做什么”彻底分离：AI 负责理解，框架负责约束，业务系统负责裁决。

生产级安全边界

五道闸门，解决领导最担心的失控、越权、说不清

Action 白名单

只允许执行注册过的业务 Action。LLM 不能自由写 SQL、不能临时发明 API、不能跳过流程。

权限 / 租户策略

统一策略层在执行前判定 subject、tenant、riskLevel。拒绝即终止并按需补偿。

人工审批挂起

高风险动作自动进入 `SUSPENDED_PENDING_REVIEW`，审批后可从同一个 runId 恢复。

Saga 逆序补偿

失败或拒绝后按已执行 Action 栈逆序撤销，草稿、审批单等副作用不留孤儿。

全链路审计

PLAN_GENERATED、ACTION_STARTED、BLACKBOARD_UPDATED、COMPENSATED 等事件全部持久化。

LLM 降级不熔断入口

LLM 失败可降级规则解释器；结构化输出畸形转澄清问题，而不是让业务流程崩溃。

为什么它值得做成平台

它不是替代工作流，也不是普通 Agent：它是业务 Action 图谱执行层

能力维度 直接接 LLM 工作流引擎 外部 Agent 平台 ActionGraph
执行路径可复现不可控可以取决于平台确定性搜索

异常改道能力靠模型猜需预画分支黑盒每步重规划

下一步建议

三周试点：用一个真实业务场景验证效率、安全、可审计

第 1 周 / 接入

选一个低风险高频场景

建议选择草稿生成、申请发起、内部审批类场景。业务团队在框架组陪跑下完成 Action 封装和 Goal 注册。

第 2 周 / 联调

接审批与持久化

跑通挂起、审批、恢复、拒绝补偿、失败补偿，安全团队检查审计链与权限策略。

第 3 周 / 灰度

小范围真实用户验证

采集操作耗时、接入人天、审批拦截率、审计完整率，输出推广决策报告。

试点成功指标

单流程耗时下降 50% 以上；接入成本不超过 3 人天；审批拦截率、补偿成功率、审计完整率均达到 100%。

请求批准

授权 1 个试点业务团队 + 框架组 1 人陪跑，启动三周真实场景试点。

战略价值

ActionGraph 的本质：把 AI 能力从“项目制”升级为公司级基础设施

每个系统注册自己的业务能力图谱，全公司共享一个安全、可审计、可恢复的 AI 执行底座。接入的系统越多，能力图谱越大，AI 能替员工完成的事就越多。

近期

试点落地，完成内部 Maven 发布、中文接入文档、审批系统对接。

中期

扩展到 2~3 个业务系统，沉淀公司统一 Action 设计规范和安全策略模板。

长期

形成内部能力图谱市场：业务能力可发现、可复用、可组合，新流程由图谱自动装配。
