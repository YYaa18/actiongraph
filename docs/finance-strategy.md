# ActionGraph 金融行业突击战略

> 定位收缩决议：Java only · 目标客户为银行/保险/证券等金融机构 · 以金融垂直场景为唯一突破口
> 本文档替代 ecosystem 全景路线中与金融无关的部分，作为后续迭代总纲。

---

## 0. 一句话定位

**ActionGraph：金融级 AI Agent 执行底座 —— 让 AI 在金融机构"持证上岗"。**

AI 能力（大模型、RPA、规则引擎）由行方自选；ActionGraph 提供监管语境下缺失的那一层：
**白名单内执行、多级人工审批、不可抵赖审计、失败自动冲正。**

## 1. 为什么金融业是唯一正确的靶子

| 金融业特征 | ActionGraph 对应资产（已实现） |
|---|---|
| 强审批文化（四眼原则、多级授权） | HumanReviewPolicy + 挂起/原子认领恢复 |
| 监管审计要求（操作留痕、可回溯、可举证） | 全链路 Trace + JDBC 持久化 + 回放 |
| 不容出错（错账必须冲正） | Saga 逆序补偿（金融语境即"自动冲正"） |
| 权限与额度刚性约束 | PolicyGuard / PermissionPolicy / 风险等级 |
| 核心系统全是 Java/Spring | Spring Boot Starter + 注解接入（半天接入实测） |
| 数据不出行（拒绝外部 SaaS agent） | 纯内部 Java 库，LLM 可对接行内私有化模型 |
| 流程确定性偏好（拒绝黑盒） | 确定性 GOAP 规划，同输入同路径，先解释后执行 |

结论：在互联网行业这些是"加分项"，在金融行业是**采购准入条件**。我们不需要改变产品本质，只需要换语言体系（补偿→冲正，审批→授权，trace→留痕）。

## 2. 竞争位势

- **不与行内大模型平台竞争**：行方已建/在建大模型底座（私有化 DeepSeek/通义/智谱）。ActionGraph 是其下游的执行治理层，叙事为"大模型平台负责想，ActionGraph 负责安全地做"。
- **不与 RPA 竞争，吃掉 RPA**：金融机构存量 RPA 资产巨大但无语义、无审批、无补偿。提供 RPA→Action 包装器，把按键精灵升级为带授权与冲正语义的受治理能力。存量 RPA 是迁移入口而非敌人。
- **与 Embabel/LangChain4j 错位**：它们是通用框架，无金融合规包、无冲正语义、无监管文档。我们不做通用市场。

## 3. 金融化能力差距清单（按优先级，交付前置）

### P0 —— 没有就进不了行（试点前必须）
1. **敏感数据脱敏**：Trace 与审批预览中的 Blackboard 可能含客户身份证号/卡号/手机号。F0-1 已引入可插拔 `DataMaskingPolicy`，在 trace append 与 HumanReviewRequest 构造处统一脱敏；挂起快照保持无损恢复并通过库表权限隔离，JDBC resume 可配置 `BlackboardTypeRegistry` 限制可恢复的业务对象类型。
2. **审计链防篡改**：F0-2 已为 TraceEvent 增加链式 SHA-256 哈希（prev_hash + 本事件哈希），JDBC 落库并提供 `TraceChainVerifier`。监管检查时可证明留痕未被修改。
3. **多级审批策略**：F0-3 已支持审批链模型，按风险等级路由经办复核→授权，复用挂起/恢复机制逐级推进；重复同级审批通过 stage index CAS 拦截。
4. **额度策略**：F0-4 已支持单笔金额提取、硬限额拒绝、超审批阈值升级审批；日累计/月累计留到 F1。

### P1 —— 投标加分项
5. **信创适配**：JDBC 仓储验证国产数据库（OceanBase / 达梦 / 人大金仓 / GaussDB）；LLM 客户端增加通义/智谱/文心适配（DeepSeek 已有）。
6. **双活/多实例验证**：挂起原子认领已支持多实例（已测），需在双活部署文档化 + 故障演练脚本。
7. **监管映射文档**：将框架能力映射到等保三级、个保法、《银行保险机构数据安全管理办法》、生成式 AI 暂行办法条款——采购评审直接引用。

### P2 —— 产品化
8. **packs-banking / packs-insurance**：预置 Action 模板与策略包。
9. **console 监管视图**：审计查询、审批收件箱、运行监控（只读优先）。

## 4. 突击场景选型（首批打穿一个）

选型标准：草稿类/申请发起类（AI 永不做终决）、高频、跨系统点击多、已有清晰审批链。

| 场景 | 行业 | 匹配度 | 说明 |
|---|---|---|---|
| **续保报价准备** | 保险 | ★★★★★ | 与参考实现 renewal 域完全同构，demo 即原型 |
| 理赔资料预审 + 赔付申请草稿 | 保险 | ★★★★★ | F1 第一刀已形成样板域：查资料→核完整性→生成赔付申请草稿→提交人工 |
| 信贷贷后检查报告 | 银行 | ★★★★☆ | 多系统取数→生成报告草稿→客户经理确认 |
| 运营调账申请 | 银行 | ★★★★☆ | 高风险高审批，补偿语义（冲正）天然匹配 |
| 反洗钱可疑交易报告草稿 | 银行 | ★★★☆☆ | 价值高但监管敏感，第二阶段再碰 |

**建议首攻：保险续保报价或理赔预审**——参考实现现成、风险可控、价值可量化（单均处理时长）。

## 5. 商业与传播路径（金融行业不是开源获客逻辑）

- **Open Core**：内核 + starter 开源（Apache 2.0，建立技术信任与开发者口碑）；金融合规包、console、信创适配商业化。金融机构偏好"开源内核 + 商业支持"形态。
- **进场路径**：① 自有/关联业务先落标杆案例 → ② 银行/保险科技子公司共建（建信金科、平安科技模式） → ③ 行业会议（金融科技大会、中国保险科技峰会）与招投标。
- **内容线**（配合而非主线）：面向行内架构师的白皮书《AI Agent 在金融机构的受控落地》＋ 监管映射文档公开版——金融行业的"内容营销"是白皮书不是博客。

## 6. 阶段计划（替代原 Phase 0–3）

- **F0（0–2 月）内核金融化**：core 1.0 冻结 + P0 四项（脱敏/防篡改/多级审批/额度）+ 监管映射文档初版。⛔ 门槛：脱敏与多级审批过安全评审。
- **F1（2–5 月）场景打穿**：保险续保或理赔预审在真实/准真实环境落地，输出单均时长、拦截率、审计完整率数据。理赔预审样板域已支持 CSV/JDBC 样本输入、脱敏视图契约、外部审批回调处理、外部审批决策输入，以及带批次/环境/限额参数/审批模式/耗时拆分的 Markdown/CSV 指标报告，并可通过 suspend/resume 模式统计审批等待。下一步映射真实业务库方言。⛔ 门槛：业务方愿意续用并出具案例。
- **F2（5–10 月）产品化**：packs-insurance / packs-banking、console 只读版、信创适配矩阵。
- **F3（10 月+）规模化**：第二家机构复制、白皮书发布、open core 分层落地。

## 7. 砍掉的东西（反目标，比目标重要）

- ❌ 游戏/机器人/边缘端口
- ❌ 跨组织 agent 事务协议（远期再议）
- ❌ 通用 MCP 公网网关（收窄为行内工具治理）
- ❌ 多语言 SDK（Java only，Python/JS 永不做）
- ❌ 通用 LLM 编排能力（让给行内大模型平台）
