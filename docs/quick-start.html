ActionGraph 快速接入指南

把你系统里已有的 Java 业务能力封装成可规划、可审计、可补偿的 Agent 能力 —— 半天完成首个接入。

这是什么

ActionGraph 是公司内部的 类型化 GOAP Agent 运行时框架。它让用户用自然语言提出业务目标，由确定性规划器（不是 LLM）从你注册的业务 Action 中搜索出执行路径，逐步执行、每步重规划，高风险动作自动进入人工审批，全过程可审计、失败可补偿。

核心分工 —— 一句话记住
LLM 只负责「理解目标、补全参数、解释结果」；规划由确定性算法完成，业务规则仍由你的业务服务裁决。LLM 永远无法绕过你注册的 Action 白名单去碰数据库或 API。

你作为业务系统接入方，只需要做三件事：把业务能力标注成 Action、注册业务目标（Goal）、处理运行结果状态。规划、执行、审批、补偿、审计都由框架完成。

架构与职责边界

用户自然语言"帮客户 C001 准备一份续约报价"
↓
Goal 解释层（LLM，可降级）识别 GoalType · 抽取参数 · 缺参追问 —— 不生成执行计划
↓
GOAP 规划器（确定性 A*/BFS 搜索）按 precondition / effect 在你注册的 Action 中搜索路径
↓
执行器（每步执行后重规划）权限闸门 · runtime guard · 人工审批挂起 · 失败逆序补偿
↓
你的业务服务真正的校验与状态变更 —— 拥有最终裁决权
↓
审计 Trace / 挂起快照 / 审批任务 / 结构化记忆内存或 JDBC 持久化

2编写 Action 适配器（核心工作量）

不需要实现任何框架接口 —— 在普通 Spring Bean 的方法上加注解即可。方法参数自动按类型从 Blackboard 取，非 void 返回值自动写回 Blackboard。

@Component
public class OrderCancellationAgentAdapter {

    private final OrderService orderService;            // ← 你已有的业务服务
    private final CancellationRequestService requestService;

    // 1. 查询类 Action：READ_ONLY，无需补偿
    @ActionGraphAction(
        id = "order.lookup",
        preconditions = "order:ORDER_ID_PRESENT",
        effects = "order:ORDER_LOADED",
        riskLevel = ActionRiskLevel.READ_ONLY
    )
    OrderRecord lookup(OrderId orderId) {
        return orderService.find(orderId);
    }

    // 2. 写入类 Action：声明风险等级，必须配补偿
    @ActionGraphAction(
        id = "order.cancellation.request.draft",
        preconditions = {"order:ORDER_LOADED", "order:ELIGIBILITY_CHECKED"},
        effects = "order:REQUEST_DRAFTED",
        riskLevel = ActionRiskLevel.MEDIUM
    )
    CancellationRequestDraft draft(OrderRecord order, CancellationEligibility eligibility) {
        return requestService.createDraft(order);
    }

    // 3. 运行期值断言：规划期不可知的业务事实，执行前由框架求值
    @ActionGraphGuard(actionId = "order.cancellation.request.draft")
    boolean canDraft(CancellationEligibility eligibility) {
        return eligibility.eligible();
    }

    // 4. 补偿（Saga）：执行链失败或审批被拒时，框架逆序调用
    @ActionGraphCompensation(actionId = "order.cancellation.request.draft")
    void voidDraft(CancellationRequestDraft draft) {
        requestService.voidDraft(draft.requestId());
    }

    // 5. 高风险 Action：requiresHumanReview = true，执行前自动挂起等审批
    @ActionGraphAction(
        id = "operations.approval.request",
        preconditions = "order:REQUEST_DRAFTED",
        effects = "order:OPS_APPROVAL_REQUESTED",
        riskLevel = ActionRiskLevel.HIGH,
        requiresHumanReview = true
    )
    OperationsApprovalRequest requestApproval(CancellationRequestDraft draft) {
        return approvalService.request(draft);
    }
}
两条硬规则
  • Action 方法必须返回具体类型（record / 实体类），返回接口或抽象类会在启动时直接报错 —— 这是刻意设计，Blackboard 按具体类型匹配。
  • condition 一律带命名空间："order:ORDER_LOADED"，命名空间用你的业务域名，避免与其他接入系统撞名。

3注册 Goal 与参数装配

Goal 声明「目标条件是什么、需要哪些参数」。注册进 GoalCatalog 后，LLM 的 prompt 会由框架自动生成，不需要手写。

@Configuration
public class OrderAgentGoalConfig {

    public static final GoalType CANCEL_ORDER = new GoalType("requestOrderCancellation");

    @Bean
    GoalCatalog goalCatalog() {
        GoalCatalog catalog = new GoalCatalog();
        catalog.register(new GoalDefinition(
            CANCEL_ORDER,
            "为指定订单发起取消申请并提交运营审批",
            new Goal("requestOrderCancellation",
                     Set.of(Condition.of("order", "OPS_APPROVAL_REQUESTED"))),
            List.of(GoalParameterDefinition.required(
                "orderId", "要取消的订单编号", "O100"))
        ));
        return catalog;
    }

    // Seeder：把解释出的参数装配成 Blackboard 初始状态
    @Bean
    GoalBlackboardSeeder orderCancellationSeeder() {
        return new GoalBlackboardSeeder() {
            @Override public GoalType goalType() { return CANCEL_ORDER; }
            @Override public void seed(GoalParameters params, Blackboard bb) {
                bb.put(new OrderId(params.get("orderId").orElseThrow()));
                bb.addCondition(Condition.of("order", "ORDER_ID_PRESENT"));
            }
        };
    }
}

4接入 LLM 解释器（自然语言入口）

设置环境变量后组装解释器。推荐挂 Fallback：LLM 网络故障时自动降级到规则解释器，入口永不不可用。

# 环境变量
DEEPSEEK_API_KEY=sk-xxxx          # 必填
DEEPSEEK_MODEL=deepseek-v4-flash  # 可选，有默认值
DEEPSEEK_BASE_URL=...             # 可选，私有网关时覆盖
@Bean
GoalInterpreter goalInterpreter(GoalCatalog catalog) {
    LlmGoalInterpreter llm = new LlmGoalInterpreter(
        DeepSeekChatClient.fromEnvironment().orElseThrow(),
        new GoalCatalogPromptRenderer(catalog),   // prompt 由目录自动生成
        new GoalInterpretationJsonParser(catalog.goalsByType())
    );
    return new FallbackGoalInterpreter(llm, myRuleBasedInterpreter); // 降级兜底
}

解释结果 GoalInterpretation 只有两种形态，处理逻辑非常简单：

  • isReady() == true → 拿到 Goal + 参数，进入执行；
  • 否则 → 把 clarificationQuestion() 返回给用户追问（如「请提供订单编号」），用户补答后带着已知参数再调 interpret(input, knownParameters)。

5运行与处理结果

@Service
public class OrderAgentService {
    private final GoalInterpreter interpreter;
    private final GoalBlackboardSeederRegistry seeders;
    private final GoapExecutor executor;       // starter 自动装配
    private final ActionRegistry registry;     // 注解 Bean 已自动注册进来

    public RunResult handle(String userInput) {
        GoalInterpretation gi = interpreter.interpret(userInput);
        if (!gi.isReady()) { /* 返回澄清问题给前端 */ }

        Blackboard bb = new InMemoryBlackboard();
        seeders.seed(gi, bb);

        return executor.run(gi.goal().orElseThrow(), bb, registry.all(), registry);
    }
}

RunStatus 状态机 —— 每个状态都要有处理分支

状态含义你该做什么
COMPLETED目标达成从 Blackboard 取结果对象返回用户
SUSPENDED_PENDING_REVIEW遇高风险动作，已挂起等审批告知用户「已提交审批」，保存 runId，审批后 resume（见下节）
HALTED_UNREACHABLE业务事实不满足（如订单已发货），无可达路径把 message() 转译成业务话术返回用户
DENIED_BY_POLICY权限或审批拒绝，已自动逆序补偿告知用户被拒原因，无需手工清理
FAILED_COMPENSATED执行失败，补偿全部成功告知失败，系统状态已还原
FAILED_COMPENSATION_INCOMPLETE执行失败且部分补偿失败必须告警人工介入，按 Trace 核对残留副作用

人工审批与挂起恢复

默认装配的 RepositoryBackedHumanReviewPolicy 把审批做成了「任务表」模型，可直接对接公司现有审批系统。默认是单级审批；金融场景可开启按风险等级路由的多级审批链。

高风险 Action 触发
→
挂起 SUSPENDED_PENDING_REVIEW
审批任务入库
→
审批系统读取 findPending()
→
审批人 decideStage(...)
→
resume(runId)
读决定继续或拒绝补偿
// 审批侧：展示待审任务（含计划预览、当前状态、Blackboard 预览）
List<HumanReviewTask> pending = humanReviewRepository.findPending();

// 审批人做出决定。expectedStageIndex 来自 task.currentStageIndex()
HumanReviewTask task = pending.getFirst();
humanReviewRepository.decideStage(task.runId(), task.actionId(), task.currentStageIndex(),
        HumanReviewDecision.APPROVED, "ops-lead", "审批通过");

// 恢复运行：原子认领 runId，trace 连续，挂起前动作的补偿链完整保留
RunResult resumed = executor.resume(runId, registry.all(), registry);
挂起恢复的语义保证
resume 会先从 SuspendedRunRepository 原子认领挂起快照；同一个 runId 被重复回调或多实例并发消费时，只有一个恢复流程能继续执行业务动作。多级审批未走完时，resume 会重新挂起并保留同一条补偿栈；审批拒绝（DENIED）同样触发补偿，不会留下孤儿草稿。 
actiongraph:
  human-review:
    risk-based-approval-chain: true   # HIGH 两级；默认 false 保持单级

敏感数据脱敏

金融场景建议开启默认正则脱敏。它只作用于两个人类可读面：TraceEvent.detail/data 与 HumanReviewRequest.blackboardPreview；SuspendedRun 的 Blackboard 快照不会脱敏，确保 resume 可无损恢复。

actiongraph:
  masking:
    enabled: true
    blocked-keys:
      - idCard
      - cardNo
      - customerName

默认规则覆盖身份证、银行卡、手机号、邮箱；blocked-keys 命中的字段会整体替换为 ***。如果需要对接行内统一脱敏服务，直接提供自己的 DataMaskingPolicy Bean 即可覆盖默认实现。

审计链防篡改

每条 TraceEvent 都会在写入前带上 prevHash 与 hash。哈希按 runId 隔离串链，计算对象是已经脱敏后的落库形态；因此监管或审计侧可以直接读取数据库里的 trace 重新计算并验证。

List<TraceEvent> events = traceRepository.findByRun(runId);
TraceChainVerifier.ChainVerification verification =
        new TraceChainVerifier().verify(events);

if (!verification.valid()) {
    log.warn("trace broken at seq {}", verification.firstBrokenSeq());
}

JDBC trace 表会自动创建或迁移 prev_hash / hash 两列。旧版本无哈希历史行不会回填，验证器会明确报告为 pre-F0 数据。

单笔额度策略

额度策略只做框架治理，不猜业务语义。接入方需要提供一个 AmountExtractor，明确说明某个 Action 在当前 Blackboard 上对应哪一笔金额。

@Bean
AmountExtractor amountExtractor() {
    return (action, blackboard) -> {
        if (!action.id().value().equals("sales.approval.request")) {
            return Optional.empty();
        }
        return blackboard.get(QuoteDraft.class)
                .map(draft -> new MonetaryAmount(draft.premium(), draft.currency()));
    };
}
actiongraph:
  limits:
    rules:
      - action-id: sales.approval.request
        currency: CNY
        hard-limit: 1000000      # 超过：DENIED_BY_POLICY，并触发已执行步骤补偿
        review-limit: 100000     # 超过：写入 amountEscalated 属性，审批链追加额度授权
  human-review:
    risk-based-approval-chain: true

有金额但币种没有配置时会保守拒绝；没有配置 AmountExtractor 的系统保持现有行为。审批任务会保存 amount、currency、amountEscalated 等 attributes，方便审批系统展示升级原因。

持久化（生产必读）

默认仓储全部是内存实现，重启即丢，仅适合开发联调。生产环境用四个 @Bean 覆盖为 JDBC 实现：

@Configuration
public class AgentPersistenceConfig {
    @Bean TraceRepository traceRepository(DataSource ds) {
        return new JdbcTraceRepository(ds);            // 审计链
    }
    @Bean SuspendedRunRepository suspendedRunRepository(
            DataSource ds,
            ActionGraphProperties properties
    ) {
        return new JdbcSuspendedRunRepository(
                ds,
                properties.getPersistence().getSuspendedRunClaimTimeout()
        );                                                  // 挂起快照，跨重启可恢复
    }
    @Bean HumanReviewRepository humanReviewRepository(DataSource ds) {
        return new JdbcHumanReviewRepository(ds);      // 审批任务
    }
    @Bean MemoryRepository memoryRepository(DataSource ds) {
        return new JdbcMemoryRepository(ds);           // 结构化长期记忆（可选）
    }
}
安全边界
挂起快照按类名 + JSON 恢复 Blackboard 对象，且为了无损 resume 不做脱敏。挂起表必须视为高敏可信输入：仅 runtime 服务账号可读写，严格控制库表权限，不要与低信任系统共库。

Action 设计规范

Action 边界是接入质量的决定因素。一个 Action 要小、明确、可测试、可补偿。

❌ 太大（拒绝）✅ 正确粒度
「处理订单」查询订单 / 校验可取消 / 创建取消申请 / 提交审批 / 通知客户经理

风险等级与补偿要求对照

Action 类型riskLevel补偿要求人审
纯查询READ_ONLY不需要—
生成草稿 / 暂存LOW / MEDIUM删除草稿或标记作废—
发起审批 / 通知MEDIUM撤回审批；通知不可回滚则发更正视情况
修改正式数据 / 付款 / 退款 / 对外发送HIGH必须有补偿，记录外部 requestId 供对账必须 true

precondition 与 runtimeGuard 怎么分

  • precondition（注解里的字符串）：「执行了某步就会成立」的符号事实，参与路径规划。例：order:ORDER_LOADED。
  • @ActionGraphGuard：要看数据的值才知道的业务事实（如 eligibility.eligible()），规划期不可知，执行前求值，不通过会自动尝试改道，无路则 HALTED_UNREACHABLE。

进阶：给主路径配备选 Action（动态补链）

给同一个 effect 提供第二个 Action，主路径被 guard 拦下时规划器会自动改道，无需任何流程编排代码。例如「查不到现有合同」时自动走「创建合同草稿」：两个 Action 产出同一个 renewal:CURRENT_CONTRACT_LOADED，下游计划完全不变。

常见坑

症状原因解法
启动报 must return a concrete typeAction 方法返回了接口/抽象类改为返回具体 record/类
运行报 Missing blackboard value for ...上游没有 Action 产出该参数类型，或返回类型与参数声明类型不一致核对类型链：上游返回的具体类型 == 下游参数类型
规划器找不到 Plan（NO_PLAN）precondition/effect 拼写不一致、命名空间漏写、或确实缺一个 Action把 condition 字符串抽成常量类统一引用
启动报 Duplicate annotated action两个 Bean 用了同一个 action idaction id 用「域.对象.动作」全局唯一命名
同类型对象互相覆盖Blackboard 同类型默认单实例多实例场景用 BlackboardKey.of(Type.class, "key") 槽位
LLM 不可用导致入口报错没挂 Fallback用 FallbackGoalInterpreter 包一层规则解释器

上线检查清单

  • ☐ 所有写入类 Action 均配有 @ActionGraphCompensation，且补偿有测试覆盖
  • ☐ 付款 / 退款 / 删数据 / 对外发送 / 改正式状态的 Action 全部 requiresHumanReview = true
  • ☐ condition 全部带本域命名空间，并抽成常量类
  • ☐ 四个仓储已替换为 JDBC 实现，挂起表写权限已收紧
  • ☐ FAILED_COMPENSATION_INCOMPLETE 已接入告警
  • ☐ 审批系统已对接 findPending() / decideStage(... currentStageIndex ...) / resume() 三件套
  • ☐ 涉及金额的 Action 已配置 AmountExtractor 与 actiongraph.limits.rules，并覆盖超硬限额/升级审批测试
  • ☐ 带真实 DEEPSEEK_API_KEY 跑通一次不含 ID 模式的自然语言输入（验证 LLM 路径真实生效）
  • ☐ 发布纪律：存在未决挂起运行时，不删除或改名已注册的 Action id
