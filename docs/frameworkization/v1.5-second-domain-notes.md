# v1.5 Frameworkization Check: Second Domain

## Purpose

This pass validates whether the runtime is reusable beyond the renewal quote example. The second domain is an order cancellation request flow, implemented without changing core runtime packages.

## Added Domain

Package: `com.actiongraph.samples.ordercancellation`

Goal: `requestOrderCancellation`

Initial input:

- `OrderId("O100")`
- `ORDER_ID_PRESENT`

Target condition:

- `OPERATIONS_APPROVAL_REQUESTED`

Action chain:

1. `order.lookup`
2. `order.cancellation.eligibility.check`
3. `order.cancellation.request.draft`
4. `operations.approval.request`

The final action is high risk and requires human review, so v2 policy behavior is exercised too. A pending review now returns `SUSPENDED_PENDING_REVIEW` and saves a suspended run snapshot.

## Integration Steps Observed

To add the second domain, the domain author had to provide:

- Domain records: id, loaded aggregate, eligibility, draft, approval request.
- Service interfaces and in-memory service implementations.
- Conditions and goal factory.
- One `Action` implementation per business capability.
- An action factory and registry helper.
- A rule-based interpreter.
- A `GoalBlackboardSeeder`.
- A small summarizer/sample.
- Flow, interpreter, and seeder tests.

No changes were needed in:

- `action`
- `planning`
- `runtime`
- `policy`
- `trace`
- `interpretation`
- `llm`

This is the strongest evidence so far that the kernel is reusable.

## Friction Points

1. Conditions now support namespaces, but migration discipline matters.

   `renewal:*` and `order-cancellation:*` are now separate keys. Annotation-based actions should use the same namespaced string form, because annotation attributes cannot refer to `Condition` constants directly.

2. Domain boilerplate is high.

   A simple four-action flow required many small files. This is acceptable for a reference implementation, but a framework should provide templates or annotation-based registration.

3. Action factories are still duplicated in the samples, but applications no longer need to copy this wiring.

   `actiongraph-spring-boot-starter` can scan ordinary Spring beans with `@ActionGraphAction`, `@ActionGraphGuard`, and `@ActionGraphCompensation`, then expose an `ActionRegistry` automatically. The samples keep explicit factories because they are pure Java samples.

4. Prompt generation is now catalog-driven for LLM-backed domains.

   Renewal registers a `GoalCatalog` and uses `GoalCatalogPromptRenderer`. The order cancellation domain currently uses a rule-based interpreter only, so it does not yet exercise LLM catalog rendering.

5. Human review now has in-process suspend/resume semantics.

   High-risk actions can stop before side effects, save a suspended run, and resume with the same run id and compensation stack. Real review still needs durable persistence and an external approval integration.

6. Blackboard now supports multiple instances per runtime class.

   Existing samples still use default single-value lookup, while multi-entity flows can use `BlackboardKey<T>` and `@BlackboardValue` for keyed values.

## Commands

The actiongraph-samples module exposes a dedicated Gradle task for the order cancellation sample:

```bash
./gradlew :actiongraph-samples:runOrderCancellationSample --args="Cancel order O100"
./gradlew :actiongraph-samples:runOrderCancellationSample --args="--approve-human-review Cancel order O100"
./gradlew :actiongraph-samples:runOrderCancellationSample --args="--shipped --approve-human-review Cancel order O100"
```

## Conclusion

The second domain was integrated without core changes. That means the runtime is no longer only a renewal sample; it is a small reusable kernel with two reference adapters.

Publishing metadata is now configured for `actiongraph-bom`, `actiongraph-core`, `actiongraph-annotations`, `actiongraph-memory`, `actiongraph-memory-jdbc`, `actiongraph-memory-spring-boot-starter`, `actiongraph-memory-jdbc-spring-boot-starter`, `actiongraph-interpretation`, `actiongraph-human-review`, `actiongraph-human-review-jdbc`, `actiongraph-human-review-spring-boot-starter`, `actiongraph-human-review-jdbc-spring-boot-starter`, `actiongraph-human-review-callback-spring-boot-starter`, `actiongraph-governance`, `actiongraph-governance-human-review`, `actiongraph-governance-spring-boot-starter`, `actiongraph-governance-human-review-spring-boot-starter`, `actiongraph-llm`, `actiongraph-llm-deepseek`, `actiongraph-persistence-jdbc`, `actiongraph-spring-boot-starter`, `actiongraph-jdbc-spring-boot-starter`, `actiongraph-console-core`, `actiongraph-console-jdbc`, `actiongraph-console-export`, `actiongraph-console-spring-boot-autoconfigure`, `actiongraph-console-api-spring-boot-starter`, `actiongraph-console-ui-spring-boot-starter`, `actiongraph-console-export-spring-boot-starter`, `actiongraph-console-jdbc-spring-boot-starter`, and `actiongraph-console-spring-boot-starter`. Core JDBC persistence now covers suspended runs, trace events, and the trace read model; structured memory and human review task persistence are split into their own JDBC modules and Spring Boot starters, so services opt into only the durable stores they need. Annotation-based action registration is split into an independent non-Spring adapter, with the Spring Boot starter reusing it for bean scanning. Structured memory/context is split into the independent `actiongraph-memory` module, with Spring defaults moved into `actiongraph-memory-spring-boot-starter` and JDBC durability moved into `actiongraph-memory-jdbc`. GoalCatalog, GoalInterpreter, interpretation results, and Blackboard seeders are split into the independent `actiongraph-interpretation` module. Repository-backed review tasks, approval chains, in-memory review storage, and callback handling are split into the independent `actiongraph-human-review` module, with JDBC durability moved into `actiongraph-human-review-jdbc`; repository-backed Spring review policy and Spring MVC callback endpoint wiring are split into separate starters. Optional masking, amount-limit rules, and rule-based permission/tenant checks are split into the independent non-Spring governance library; amount review attributes and risk-based approval routing are split into `actiongraph-governance-human-review`. LLM interpretation is split into provider-neutral interpretation support plus the DeepSeek provider. Human review has a repository-backed external decision path. Blackboard supports keyed multi-instance values. v3 dynamic repair planning is demonstrated through a renewal fallback action without core planner/executor changes. v4 structured memory/context is available through `MemoryRepository` and `MemoryContextLoader`. The read-only Console is split into a reusable core query service, a JDBC read-model adapter, a pure Java CSV/JSONL audit export component, shared Spring service auto-configuration, optional JSON API starter, optional UI starter, optional export endpoint starter, optional JDBC Spring repository adapter, and a compatibility aggregate starter.
